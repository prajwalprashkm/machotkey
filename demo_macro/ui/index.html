What each JSON field means

action_ops_per_sec is sum(ops) ÷ sum(timed workload µs) over the metrics window—throughput of the measured work only, not limited by capture pacing between callbacks.

  • callbacks_per_sec — Completed capture callbacks per second (wall time for the window).
  • callback_raw_fps — Same time base as above, but the numerator adds skipped frame indices in the window (stream pressure / drops).
  • latency_ms — Average time from frame capture timestamp to callback start (same window).
  • action_label — Name of the primary timed API for this phase (e.g. find_color), when applicable.
  • action_ops_per_sec — Primary workload ops per second from summed workload time (see note above).
  • action_avg_us_per_op — Window average: sum(workload µs) ÷ sum(ops).
  • workload_ops_last_batch, workload_us_per_op_last_batch, workload_total_ms_last_batch — The last capture callback in the window only (same as the HUD “last batch” line).

Tuning (→ config_user.lua)

What each phase does

  • idle — capture + canvas only (baseline).
  • color — find_color on the full frame until the adaptive time budget for TARGET_FPS is used (see tuning).
  • ocr_fast / ocr_accurate — one full-frame recognize_text per frame (timing = that single pass).
  • input — mouse.get_position in the same adaptive budget (no synthetic clicks).
  • opencv — one full-frame pipeline (Mats + match_template) after you press OK on the CPU warning; until then this phase is idle workload.
  • fs — read_all on config.lua until the budget is used.

OpenCV is CPU-heavy

OpenCV work (template matching, converting large buffers to Mat, etc.) can use a lot of CPU for sustained periods (over 100% usage of a single core).

In the main Machotkey app, open Settings and edit Runtime resource limits: CPU throttle-at and CPU hard-stop-at (also cpu_throttle_percent / cpu_kill_percent in your project manifest.json).

Keep the hard-stop (kill) limit much higher than the throttle threshold, and use the throttle to control how much CPU the macro may use. If the kill limit is too low or too close to normal OpenCV load, the runner can sit at or above the kill line and the macro will be stopped after about 3 seconds.

OpenCV work in this demo does not start until you press OK. The automated suite also waits here before advancing to the next phase.
