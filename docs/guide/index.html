Getting Started

Machotkey is a high-performance native macOS macro scripting application. Write macros in Lua (LuaJIT), automate keyboard and mouse input, capture and analyze the screen with OpenCV, and build UI panels through embedded WebKit views.

Overview

Machotkey splits privileges between two processes to keep scripting constrained while the host retains the capabilities needed for platform actions:

machotkey is the main host process — it manages the UI, elevated OS capabilities, and mediates privileged operations on behalf of macros.

macro_runner is a sandboxed child process that runs your Lua scripts and communicates with the host over IPC and shared memory. Malicious or buggy scripts are contained here.

Macros are directories containing at least a manifest.json file; however, virtually all macros also contain Lua files and other assets. The API Reference covers every function available to scripts. A project's manifest.json declares its required/optional permissions, file access, runtime limits, and other metadata before execution begins. Machotkey also offers a minimal quick-script mode where users can run single-file Lua macros; however, this is not meant for production and is solely for quick testing, debugging, and exploration.

Requirements
Machotkey targets Apple Silicon Macs running macOS Sonoma or later. Requirements differ slightly depending on whether you install from a release DMG or build from source.
ItemDetail
Platform macOS 14.0 (Sonoma) or later
CPU Apple Silicon (arm64). Intel Macs are not supported.
RAM No hard minimum. The application itself typically uses under 200 MB; actual usage depends on the macros you run.
Displays Multiple displays supported for capture and automation
Toolchain (build from source only) Xcode Command Line Tools or full Xcode (Clang + macOS SDK)
CMake (build from source only) Not included with Xcode tools — install via Homebrew (brew install cmake) or from cmake.org
Git (build from source only) Included with Xcode Command Line Tools. Submodule support required — dependencies are vendored under project/libs/
Builds have been tested on an Apple M2 MacBook Air with 8 GB RAM. Other Apple Silicon Macs should work. The first source build takes longer than usual because OpenCV is compiled from source — expect it to be CPU- and memory-intensive.
Installation
There are two ways to install Machotkey. The DMG release is the easiest — no dependencies required. Building from source gives you the latest changes but requires a C++ toolchain.
1
Download the DMG
Grab the latest release from the GitHub Releases page. Download the .dmg file.
2
Open and install
Double-click the DMG, then drag Machotkey.app into your Applications folder.
3
Allow the app to run
Releases are unsigned. The first time you open Machotkey, macOS will block it. Go to System Settings → Privacy & Security, scroll down, and click Open Anyway. You only need to do this once.
1
Install Xcode Command Line Tools
Provides Clang, the macOS SDK, make, and git. Run:
xcode-select --install
Alternatively, install the full Xcode app from the Mac App Store, which includes everything the command line tools provide.
2
Install CMake
CMake is not included with Xcode Command Line Tools and must be installed separately. The easiest way is via Homebrew:
brew install cmake
If you prefer not to use Homebrew, download the macOS installer directly from cmake.org/download and follow the instructions there to add cmake to your PATH.
3
Clone the repository with submodules
Dependencies (OpenCV, LuaJIT, sol2, ImGui, etc.) are tracked as git submodules. Clone with --recurse-submodules to fetch them in one step:
git clone --recurse-submodules https://github.com/prajwalprashkm/machotkey.git
cd machotkey
If you already cloned without the flag, run git submodule update --init --recursive from inside the repo.
4
Build with CMake
From the repository root, configure and build:
mkdir build && cd build
cmake ..
cmake --build . -j 8
Or as an equivalent out-of-tree build:
cmake -S . -B build
cmake --build build -j 8
The -j 8 flag sets the parallel job count — adjust it to match your CPU core count. The first build compiles OpenCV from source, which is significantly slower than subsequent incremental builds.
5
Run the app
After a successful build, the app bundle is at:
build/bin/machotkey.app
Source builds are unsigned. macOS may require you to allow the app in System Settings → Privacy & Security the first time you open it.
Machotkey requires several macOS permissions to function — Screen Recording, Accessibility (for keyboard and mouse control), and Input Monitoring (for listening to key combos). Grant these in System Settings → Privacy & Security when prompted. Individual macros declare which permissions they need in their manifest.json.
Your first macro

Macros are directories containing at least a manifest.json file; however, virtually all macros also contain Lua files and other assets. Use the API Reference to see the available API features and write your macro. The manifest.json declares the macro's required/optional permissions, file access, runtime limits, and other metadata before execution begins. Note that while a minimal quick-script mode is available and built into the application, it is not recommended and is only meant for quick exploration. Do not use it for production. For development, an external IDE such as VSCode with Lua language support is recommended.


1
Create a project folder
mkdir my-macro && cd my-macro
2
Write a manifest.json
Declare the entry point and the permissions your macro needs. Only entry is required.
{
  "name": "My Macro",
  "version": "1.0.0",
  "entry": "main.lua",
  "permissions": [
    { "name": "keyboard_control" },
    { "name": "mouse_control" }
  ]
}
See the manifest reference for the full list of fields including file access rules and rate limits.
3
Write your entry file
Create main.lua. The entire API is available under the global system table.
-- Press Cmd+Shift+H to type "Hello, world!"
system.on_key("cmd+shift+h", function()
  system.keyboard.type(
    system.keyboard.TypeMode.STRING,
    "Hello, world!"
  )
end)
4
Open the project in Machotkey
Launch machotkey.app and open your project folder. Machotkey will read the manifest, prompt for any permissions not yet granted, and start running your macro.
Performance
Figures below are author-measured under a real macro workload (automating a CPU/GPU-heavy game with continuous screen capture and input). Treat them as representative, not universal — resolution, refresh rate, number of active capture displays, and script logic all affect your results.

Test hardware: Apple M2 MacBook Air, 8 GB unified memory.

Frame latency measurement: Latency is measured between the Mach timestamp attached to a frame by ScreenCaptureKit and the timestamp when that same frame becomes observable in Lua — isolating capture-to-delivery overhead rather than a synthetic microbenchmark.

High-DPI note: "Full-frame" color search figures use a 2× Retina buffer at 2214×3420. Other resolutions scale cost proportionally.

AreaResult
Frame pipeline latency Under 0.2 ms typical — ScreenCaptureKit timestamp to Lua-observable frame, measured under the game automation workload
Capture throughput Typically holds 90%+ of the display refresh rate; often 200+ FPS where the display supports it. Ceiling follows display refresh rate and system load.
Color search ~2 ms for a full-frame scan at 2214×3420 using the optimized search path (content-dependent)
Input automation Low-latency synthesized keyboard and mouse events, suitable for interactive automation
Application RAM overhead Typically under 200 MB for the application itself; near zero for the macro runner process
Sandboxing Seatbelt-style sandbox on the macro runner — buggy or malicious scripts are contained without affecting the host
